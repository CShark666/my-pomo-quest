### Ідея:

Планер задач з системою Pomodoro та елементами гейміфікації.

#### Фічі:

- Планування задачі з визначеними часовими рамками
  - створення задачі
    - назва
    - категорія
    - загальний запланований час,
    - часовий помодоро інтервал
    - загальна кількість інтервалів
    - час на короткий відпочинок
    - час на довгий відпочинок
  - редагування задачі:
    - назва
    - категорія
    - час на відпочинок
  - зміна статусу
- Статистика виконаних задач протягом дня/тижня/місяця
  - інфографіка виконаних та невиконаних задач на день
  - стріки за щоденну активність
- Елементи гейміфікації
  - досвід за виконані задачі
  - додатковий досвід за стрік виконаних задач
  - механіка вихідних
  - різна складність задач - різна кількість досвіду

### Pomo-quest:

Планування задачі із визначеними часовими рамками та поділом на рівні часові етапи за методом Pomodoro.

Задача (Pomo-quest) представляється у вигляді гейміфікованого квесту (pomo-quest), який містить:

- HP-бар, який відображає загальний прогрес виконання (1 ячейка бару - 1 етап задачі)
- Таймер, показує прогрес поточного етапу
- ~~Панель опису, містить всю інформацію про задачу (pomo-quest) та її статус~~

\*Наприклад, задача - навчання.

- Загальний час: 2 години
- Тривалість одного етапу: 20 хвилин
- Кількість етапів: 6
- Коротка перерва: 5 хвилин
- Довга перерва: 10 хвилин

Кожен часовий етап представляється як окрема кімната данжу або частина НР боса.
Після проходження всіх 6 "данжів" користувач отримує перемогу - задача виконана.

###### При створенні задачі користувач може вказати

- назву/ціль
- категорію (тег)
- час на задачу
- час інтервалу або кількість інтервалів (якщо час то кількість обчислюється автоматично, якщо кількість то відповідно час перерахується)
- час перерв, або без перерв

###### Після створення задачі користувач може:

- видалити її
- редагувати
  - назву
  - категорію (тег)
  - час та кількість інтервалів
  - час перерв
- розпочати виконання (статус задачі зміниться на "в процесі виконання")

###### В процесі виконання задачі користувач може:

- скасувати її (статус задачі зміниться на "скасована")

###### Під час перерви користувач може:

- пропустити

Після виконання задачі її статус міняється на "виконано", а користувач отримує досівд, який зберігається в рівні його профілю.

###### Статуси задачі:

- запланована/підготовка
- в процесі виконання
- перерва
- виконана
- скасована

###### Перерви:

Після завершення кожного етапу користувач отримує перерву для відпочинку. Перерви чергуються: коротка (5 хвилин) → довга (10 хвилин) → коротка → довга і так далі.

###### Категорія (теги)

До кожної задачі користувач може додати тег, зі списку стандартних, або створити власний тег, для категоризації задач. Теги можна змінювати при створенні, під час виконання та після виконання задачі.

### Технології:

##### frontend:

- React
- Vite
- Js/CSS/HTML

##### backend:

- .NET WEB API
- Entity Framework Core
- SQLite

### Архітектура:

###### Backend

відповідає за:

- валідацію даних
- зберігання даних
- обчислення даних (час, статистика)
- зберігання кастомних категорій користувача
- керування таймером

###### Frontend

відповідає за:

- візуалізацію даних користувача
- збір даних від користувача
- рендер таймера на основі даних з backend

###### Принцип роботи таймера

- Таймер існує лише на backend
- Frontend не зберігає стан таймера, та:
  - періодично запитує (polling) актуальний стан із backend
  - відображає отримані дані

###### API протокол

- Створення квеста
  - Вхід: запит на створення та дані квеста
  - Вихід: або створений квест або нічого
- Передача активного квеста клієнту
  - Вхід: нічого
  - Вихід: квест
- Скасування квеста
  - Вхід: нічого
  - Вихід: нічого
- Пропуск перерви
  - Вхід: нічого
  - Вихід: нічого
- Примусово завершити перерву
  - Вхід: нічого
  - Вихід: нічого
